# VCS 프로젝트 개요 (기준 문서)

> VCS 프로젝트의 배경/구조와 확정된 요구사항을 누적 기록하는 **기준 문서**다.
> 여기 쌓이는 요구사항(7장)은 [DESIGN.md](DESIGN.md)(고객 요구사항 명세서 자동화 시스템)의
> 실제 입력 예시로도 사용된다.

## 1. 사업 배경

- **SEMES**(우리)가 **삼성전자 EDS 설비공정그룹**(고객사)과 진행하는 프로젝트.
- 목표: 반도체 **후공정 EDS 라인**에 **AMR 자동화 시스템**을 구축하는 것.

## 2. Task (AMR이 수행하는 이송 작업)

AMR이 옮기는 대상은 **Probe Card**이며, Stocker와 Prober 사이를 왕복한다.

| 방향 | 설명 |
|---|---|
| Stocker → Prober | EDS 테스트를 진행해야 하는 Probe Card를 투입 |
| Prober → Stocker | EDS 테스트가 끝난 Probe Card를 회수 |

newAMOS가 VCS에게 "Stocker에서 PCard 빼서 Prober에 넣고, EDS 테스트가 끝나면 다시 Prober에서 빼서 Stocker에 넣어라"라고 명령하면(Job), VCS는 이를 Task 단위로 쪼갠다. 왕복 1회(Job 1건)는 아래처럼 8개 Task로 구성된다.

| seq | taskType | 설명 |
|---|---|---|
| 1 | MOVE_TO_UNLOAD | AMR이 Stocker까지 이동 |
| 2 | UNLOAD | Stocker에서 PCard를 빼서 AMR에 싣기 |
| 3 | MOVE_TO_LOAD | AMR이 Prober까지 이동 |
| 4 | LOAD | Prober에 PCard 투입 (→ EDS 테스트 시작) |
| 5 | MOVE_TO_UNLOAD | (테스트 종료 후) AMR이 Prober까지 이동 |
| 6 | UNLOAD | Prober에서 PCard를 빼서 AMR에 싣기 |
| 7 | MOVE_TO_LOAD | AMR이 Stocker까지 이동 |
| 8 | LOAD | Stocker에 PCard 반납 |

## 3. 시스템 계층 구조 (3-tier)

```
newAMOS (삼성전자, 상위 시스템 · MES 역할)
   │  Task 할당
   ▼
VCS (세메스, 중간 계층)
   │  유휴 상태인 AMR에게 Task 할당 + 출발지→목적지 경로탐색
   ▼
AMR (실제 이송 로봇, 다수 대)
   │  상태값을 VCS에 지속 보고
   ▼
VCS ── 고장 발생 시 newAMOS로 보고
```

- **newAMOS**: 삼성전자 소유의 상위 시스템. MES 역할을 하며 전체 Task를 관리하고 VCS에 Task를 할당한다.
- **VCS**: newAMOS로부터 Task를 받으면, 현재 작업 중이지 않은(유휴) AMR을 골라 해당 Task를 할당한다. 출발지 노드 → 목적지(dest) 노드로 경로 탐색을 수행해 AMR이 Probe Card를 옮기게 한다. AMR로부터 상태값을 계속 수신·점검하고, 고장 발생 시 newAMOS로 보고한다.
- **AMR**: 실제로 Probe Card를 물리적으로 이송하는 로봇. 다수 대가 동시에 운용된다.

## 4. VCS 내부 구성 — UI vs APP

VCS는 **UI**와 **APP**으로 나뉜다.

- **VCS UI**: 고객사(삼성전자)가 주로 사용하는 화면. 로직은 없고 표시/조작 전담.
  - 실제 스펙: Java 1.7.0_80 (32bit) + SWT 데스크톱 앱. 구버전이라 REST/API 통신이 사실상 불가 — 오류 알림 등은 App이 DB에 쓴 값을 UI가 DB 폴링으로 읽어와 표시한다.
- **VCS APP**: 실제 로직을 전담하는 백엔드.
  - 실제 스펙: Java 21 기반 MSA, 총 8개 모듈.

## 5. VCS APP — MSA 모듈 구성 (8개)

| # | 모듈 | 역할 |
|---|---|---|
| 1 | **Operation** | 가장 핵심적인 모듈. AMR과 주기적으로 통신(상태값 수신·점검)하고 Task를 관리한다. 다른 모듈들이 서로 통신할 때도 Operation을 거친다. |
| 2 | **HostInterface** | 상위 시스템 newAMOS와 통신하는 인터페이스. |
| 3 | **PathSearch** | 출발지 노드와 목적지 노드가 정해지면 다익스트라 알고리즘으로 경로를 계산해 알려준다. |
| 4 | **Watchdog** | 시스템/프로세스 상태를 감시하는 워치독 (헬스 체크·이상 감지). |
| 5 | **MapUpdater** | 라인 맵 업데이트 전담. 상시 동작이 아니라 필요 시 일시적으로 동작한다. |
| 6 | **ParameterManagement** | 시스템 파라미터 관리 모듈. |
| 7 | **NATS** | MSA 구조 내 모듈 간 통신을 담당하는 메시지 브로커. |
| 8 | **JobAssign** | AMR에게 실제 작업(Task)을 할당하는 모듈. |

## 7. 요구사항 상세 (누적)

> 개발하면서 확정되는 VCS 요구사항을 여기에 계속 누적한다.

### 7.1 태스크 전개와 전환

- 태스크 스텝은 **ASSIGNED → MOVE_TO_LOAD → MOVE_TO_UNLOAD → LOAD → UNLOAD** 총 5단계로 전환된다.
- 태스크 각 스텝별 진행에 필요한 **필수 요건이 충족된 상태인 경우에 한하여**, AMR 사용자가 메뉴를 통해 몇 단계 이전 스텝으로 돌아가 태스크를 다시 전개하거나, 몇 단계를 건너뛴 스텝에서 태스크를 다시 전개할 수 있어야 한다.
  - 단, AMR 사용자 메뉴를 통해 이전 스텝으로 돌아가는 기능은 제공하지 않는다 (이전 스텝 복귀는 VCS 기능).
- 주기적으로 수신받는 Status 정보로 LOAD 혹은 UNLOAD 작업의 수행 여부를 판단하며, 다음 단계부터 작업을 재개하도록 지시한다.
- **비고**: 태스크 각 스텝별 진행에 필요한 "필수 요건"을 정의해야 한다. (현재 미정의)

### 7.2 태스크 할당 해제와 삭제

- 사용자 메뉴를 통해 특정 태스크를 할당 해제 또는 삭제할 수 있게 한다.

**비고**
- 태스크 할당 해제는 **자재 미보유 상태에서만** 허용한다.
- 태스크 삭제 시 사용자로부터 **태스크 재생성 여부를 확인**받고, 선택에 따라 동일한 태스크를 재생성할 수 있게 한다. 재생성 없이 삭제하는 경우, 외부 시스템(newAMOS)의 태스크도 함께 삭제할 수 있도록 **메시지 형식을 정의**한다.
- **자재 보유 상태**에서의 태스크 삭제는 자재의 수동 반송을 전제하며, 태스크 삭제와 동시에 외부 시스템에 **태스크 수행 완료를 보고**한다.
- 태스크 삭제 시 사용자 **코멘트를 입력**할 수 있게 하여 이력 관리가 가능하게 한다.
- 작업 중인 태스크 삭제 시 동작:
  - AMR이 **이동 중**일 때는 태스크 삭제 가능.
  - **이적재(적재/하역) 작업 중**에는 삭제 불가.
  - 이적재 시작 후 태스크 삭제 시, AMR은 **NACK 처리**한다.

### 7.3 태스크 진행 상황 및 완료 보고

- 태스크 스텝 변경 시마다 외부 시스템에 진행 상황을 보고한다.

**비고**
- 태스크 진행 상황 및 완료 보고를 위한 **메시지 형식을 정의**한다.
- 진행 상황 보고 항목에 **AMR ID, AMR Status, 현재 스텝, 현재 위치, 현재 시각**을 반드시 포함한다.
- 완료 보고에 필요한 항목은 별도로 정의한다.
- 통신 끊김으로 보고 메시지가 VCS로 전달되지 않더라도 태스크는 멈추지 않고 계속 진행하며, 통신 연결 복구 시 **소급 보고**한다. 이때 **보고 시각**과 **실제 상태 전환 시각**을 구분하여 표시한다.
- AMR에서는 통신 연결 복구 시 소급 보고하지 않고, VCS에서 작업 완료 처리한다.
