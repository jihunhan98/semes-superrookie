# VCS 알람 트러블슈팅 어시스턴트 — 초기 설계서 (프로토타입)

## 1. 배경 및 목적

SEMES VCS(Vehicle Control System)는 AMR(무인이송로봇) 여러 대를 제어해 Prober Card를 Stocker와 Prober 사이로 이송하는 시스템이다. 운영 중 AMR/구간별로 다양한 알람·장애가 발생하는데, 원인 파악과 조치가 담당자 개인의 경험(특히 시니어의 암묵지)에 크게 의존한다. 신입/저연차 담당자는 참고할 트러블슈팅 매뉴얼이 부실해 매번 시니어에게 물어봐야 하고, 과거에 이미 처리된 유사 장애도 다시 처음부터 원인을 찾는 경우가 많다.

이 프로젝트는 **과거 장애 이력(DB) + SOP 문서**를 근거로 삼아, 알람 발생 시 "과거에 이런 알람이 있었고, 이렇게 조치했다"를 자연어로 요약해주는 RAG(Retrieval-Augmented Generation) 기반 어시스턴트를 만드는 것을 목표로 한다.

**VCS의 배차/경로/충돌회피 등 제어 로직 자체는 건드리지 않는다.** 이 프로젝트는 어디까지나 기존 VCS 운영을 보조하는 독립적인 웹 서비스다 — 안전-critical한 제어 루프에 개입하지 않으므로 1인·저연차 개발자가 완주하기에 현실적인 범위다.

## 2. 운영 환경 및 제약

- **폐쇄망(에어갭) 환경**: 외부 인터넷/외부 AI API 사용 불가. 모든 구성요소는 사내망 안에서 동작.
- AI 모델은 Hugging Face 오픈소스 모델을 다운로드해 로컬 서빙 (이전 스파이크에서 검증한 Ollama + qwen2.5:7b-instruct 조합 재사용 가능).
- 장애 이력은 이미 사내 DB에 쌓여 있음 — 신규로 만들 필요 없이 기존 데이터를 활용하되, 부족한 필드(조치 결과 등)는 보강용 샘플 데이터를 추가해도 됨.
- SOP 문서는 현재 존재하지 않음 — 이 프로젝트 진행 중 직접 작성해 RAG 근거 자료로 사용.

## 3. 시스템 아키텍처

```
[React (Next.js)] ── 알람 검색/질의 UI, 답변 + 근거(출처) 표시
        │ REST
[Spring Boot 백엔드] ── 장애 이력 DB 조회, 질의 이력 저장, AI 서버 호출
        │ REST
[FastAPI AI 서버] ── 임베딩 생성 + 벡터 검색(장애이력 + SOP) + LLM 프롬프트 구성/호출
        │
[Ollama — 로컬 LLM]      [벡터 저장소 — 장애이력/SOP 임베딩]
```

- 기존 SEMES 프로젝트와 동일하게 메인 서비스(Spring)와 AI 추론(Python)을 분리 — AI 쪽 모델 교체가 메인 서비스에 영향을 주지 않도록 함.
- AI 서버는 무상태 API: "질문 + 컨텍스트" → "답변 + 근거 목록" 구조화 출력만 담당.

### 3.1 요청 처리 흐름 (End-to-End)

질의 한 번이 답변으로 돌아오기까지 6단계를 거친다.

| # | 단계 | 담당 컴포넌트 | 처리 내용 |
|---|---|---|---|
| 1 | 입력 | Frontend | 알람코드 또는 자연어 상황 설명을 입력 |
| 2 | 접수 및 이력 저장 | Backend (Spring) | AI 서버로 전달 + `QUERY` 테이블에 질의 원문 선기록 |
| 3 | 임베딩 + 벡터 검색 | AI 서버 (FastAPI) | 질의 임베딩 → `FAULT` Top-K, `SOP` Top-K 유사도 검색 |
| 4 | 프롬프트 구성 + LLM 호출 | AI 서버 → Ollama | 검색된 컨텍스트 + 질문을 로컬 모델에 전달 (외부 API 없음) |
| 5 | 구조화 답변 생성 | AI 서버 | 원인 후보 / 권장 조치 / 근거를 구조화 출력. 근거 없으면 "확인 필요" |
| 6 | 화면 표시 + 피드백 | Frontend → Backend | 답변·근거 표시, 조치 결과(해결/미해결)를 `QUERY`에 반영 |

```
1) POST /api/troubleshoot  { "query": "3번 AMR이 Stocker A 앞에서 자꾸 멈춰요" }

2) INSERT INTO QUERY (id, raw_text, asked_at)

3) embed(query) → top_k(FAULT, k=5)  +  top_k(SOP, k=3)

4) model: qwen2.5:7b-instruct  ·  context: FAULT×5 + SOP×3

5) { "cause": [...], "action": [...], "evidence": ["FAULT-2025-114", "SOP-007"] }

6) UPDATE QUERY SET feedback = 'resolved' WHERE id = ...
```

## 4. 데이터 모델

| ID | 설명 |
|---|---|
| `FAULT-{연도}-{번호}` | 개별 장애/알람 이력. 기존 DB 데이터 재사용 |
| `SOP-{번호}` | SOP 문서를 조항 단위로 쪼갠 청크 |
| `QUERY-{번호}` | 사용자 질의 + 응답 + 피드백 이력 (신규 테이블) |

**FAULT** 주요 필드: 발생시각, AMR ID, 알람코드, 알람메시지, 발생 구간(예: Stocker A ↔ Prober B), 조치 내용, 조치자, 해결까지 걸린 시간, (부족하면 조치 내용/결과 필드는 보강 필요 — 5번 참고)

**SOP** 주요 필드: 제목, 카테고리(알람코드 그룹), 본문, 관련 알람코드 리스트

**QUERY** 주요 필드: 질의 원문, 검색된 FAULT/SOP 근거 ID 목록, LLM 응답, 사용자 피드백(해결됨/안됨), 질의 시각 — 2.1(피드백 루프)의 기반 데이터

## 5. 기능 목록

### 5.1 핵심 모듈 (Must-have)

| # | 기능 | 설명 |
|---|---|---|
| A | 알람 기반 RAG 검색 | 알람코드/메시지 입력 → 유사 과거 장애(FAULT) + 관련 SOP를 검색해 원인 후보·조치 방법을 요약. 상세 5.1.1 |
| B | 자연어 질의 인터페이스 | 알람코드 없이 자유 텍스트("3번 AMR이 Stocker 앞에서 자꾸 멈춰요")로도 동일 파이프라인 사용. 상세 5.1.2 |
| C | 근거 출처 표시 | 답변과 함께 참고한 FAULT-ID/SOP-ID를 표시해 신뢰성·추적성 확보 (환각 방지 핵심 장치). 상세 5.1.3 |

#### 5.1.1 상세 — A. 알람 기반 RAG 검색

- 입력: 알람코드 또는 알람메시지 텍스트.
- 처리: 입력을 임베딩 → 벡터 유사도로 FAULT 이력 Top-K, SOP 청크 Top-K 검색 → 검색된 컨텍스트를 LLM에 넣어 "가능한 원인 / 권장 조치 / 참고할 만한 과거 사례"로 구조화 요약.
- 과거 사례가 전혀 없는 신규 알람이면 "유사 사례 없음"을 명시하고 SOP만으로 답변하거나, 답변 불가를 솔직히 표시 (없는 사례를 지어내지 않는 것이 핵심 요구사항).

#### 5.1.2 상세 — B. 자연어 질의 인터페이스

- 담당자가 알람코드를 모르거나 여러 알람이 겹친 상황을 자연어로 설명해도 같은 RAG 파이프라인을 태운다.
- 자유 텍스트 → (필요시) 관련 알람코드 후보를 LLM이 먼저 추정 → 그 알람코드로 A와 동일한 검색/요약 수행.

#### 5.1.3 상세 — C. 근거 출처 표시

- 답변 화면에 "이 답변은 FAULT-2025-114, FAULT-2025-220, SOP-007을 근거로 생성됨"처럼 원본 링크를 함께 노출.
- 담당자가 원문(원래 장애 이력 상세, SOP 원문)을 즉시 대조 확인할 수 있게 함 — LLM 답변을 그대로 믿지 않고 검증 가능하게 만드는 것이 목적.

### 5.2 확장 기능

| # | 기능 | 설명 |
|---|---|---|
| 1 | 조치 결과 피드백 루프 | 답변대로 조치한 결과(해결/미해결)를 남기면 향후 검색 랭킹에 반영. 상세 5.2.1 |
| 2 | SOP 문서 관리 화면 | 담당자가 SOP를 직접 추가/수정하고 재임베딩. 상세 5.2.2 |
| 3 | 알람 트렌드 대시보드 | 최근 급증한 알람, 반복 발생 Top N 집계. 상세 5.2.3 |
| 4 | 장애 처리 후 SOP 초안 자동 생성 | 신규 장애의 원인/조치 기록을 LLM이 SOP 초안 형태로 요약 제안. 상세 5.2.4 |
| 5 | 유사 알람 클러스터링 | 과거 장애 이력을 임베딩 기반으로 군집화해 미분류 패턴 발견. 상세 5.2.5 |

#### 5.2.1 상세 — 1번. 조치 결과 피드백 루프

- 답변을 본 담당자가 "이 조치로 해결됨" / "해결 안 됨"을 남기면 QUERY 테이블에 기록.
- 해결됨으로 표시된 사례는 이후 유사 검색에서 가중치를 높여 우선 노출 (단순 최신순보다 "실제로 통했던 사례" 우선).

#### 5.2.2 상세 — 2번. SOP 문서 관리 화면

- SOP는 이 프로젝트 기간 중 처음부터 작성해야 하므로, 담당자(개발자 본인 포함)가 웹에서 직접 SOP를 추가/수정할 수 있는 화면이 필요.
- 저장 시 자동으로 청크 분할 + 재임베딩해 검색 대상에 즉시 반영.

#### 5.2.3 상세 — 3번. 알람 트렌드 대시보드

- 최근 N일 알람 발생 빈도를 알람코드/AMR/구간별로 집계.
- 급증한 알람이나 반복 발생 알람을 하이라이트해 "SOP 우선 작성 대상"을 자연스럽게 추천하는 효과도 있음 (4번과 연결).

#### 5.2.4 상세 — 4번. 장애 처리 후 SOP 초안 자동 생성

- 신규 장애를 처리한 담당자가 원인/조치를 자유 텍스트로 기록하면, LLM이 이를 SOP 포맷(제목/카테고리/본문/관련 알람코드)의 초안으로 변환 제안.
- SOP가 아예 없는 상태에서 시작하는 프로젝트 특성상, SOP 자산을 수작업 없이 축적하는 핵심 장치.
- 초안은 자동 확정하지 않고 담당자 검토 후 2번 화면에서 확정 (AI 결과는 초안이라는 원칙, 이전 프로젝트와 동일).

#### 5.2.5 상세 — 5번. 유사 알람 클러스터링

- 과거 FAULT 이력을 임베딩 기반으로 군집화해, 알람코드는 다르지만 실제로는 같은 근본 원인을 공유하는 그룹을 찾아냄.
- SOP 미작성 상태에서 "이런 알람 그룹부터 SOP를 만들면 커버리지가 크게 오른다"는 우선순위 판단 근거로 활용.

## 6. 비기능 요구사항

- 폐쇄망 환경에서 전 구성요소 동작 (모델/패키지는 사전 다운로드 후 반입).
- 실제 알람 대응 상황에서 쓰이므로 응답 지연이 크면 안 됨 — 목표 응답 시간(예: 수 초 이내)을 프로토타입 단계에서 실측 후 확정.
- AI 서버는 GPU 자원이 제한적인 공용 워크스테이션에서도 동작 가능하도록 경량/양자화 모델을 기본 옵션으로 고려 (이전 스파이크의 qwen2.5:7b-instruct 실측치 재사용).
- LLM이 근거 없는 답변을 지어내지 않도록(환각) 프롬프트/검증 장치 필요 — 근거(FAULT/SOP ID)가 하나도 검색되지 않으면 "확인 필요" 답변으로 유도.

## 7. 오픈 이슈 (확인 필요)

- [ ] 장애 이력 DB의 실제 스키마/접근 권한 (읽기 전용 계정으로 충분한지, 개인정보/보안 마스킹 필요 필드가 있는지)
- [ ] 임베딩 벡터 저장 방식 — 사내 Oracle DB가 23ai(벡터 검색 내장)인지 그 이하 버전인지에 따라 별도 벡터 저장소(FAISS 등 로컬) 필요 여부 결정 (이전 프로젝트 9.1과 동일한 이슈)
- [ ] 초기 SOP 작성 범위 — 전체 알람코드를 다 커버할 수 없으므로, 3번(트렌드 대시보드)으로 빈도 높은 알람부터 우선 작성할 대상 선정
- [ ] 워크스테이션 GPU 사양 (모델 크기/양자화 수준 결정에 직결)

## 8. 향후 진행 순서 (제안)

1. 장애 이력 DB 샘플 추출 + SOP 5~10건 직접 작성 (우선순위 높은 알람 위주)
2. AI 모듈(A) 스파이크 — 소규모 샘플로 RAG 파이프라인(임베딩→검색→LLM 요약)이 실제로 쓸만한 답변을 주는지 검증 (이전 프로젝트 스파이크 방식 재사용)
3. 데이터 모델/스키마 확정, Spring 백엔드 설계
4. 프론트엔드 와이어프레임 (질의 UI + 근거 표시)
5. 확장 기능(1~5) 순차 적용, 피드백 루프(1번)부터 우선
