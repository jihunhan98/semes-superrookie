# VCS 알람 트러블슈팅 어시스턴트 — 초기 설계서 (프로토타입)

## 1. 배경 및 목적

SEMES VCS(Vehicle Control System)는 AMR(무인이송로봇) 여러 대를 제어해 Prober Card를 Stocker와 Prober 사이로 이송하는 시스템이다. 운영 중 AMR/구간별로 다양한 알람·장애가 발생하는데, 원인 파악과 조치가 담당자 개인의 경험(특히 시니어의 암묵지)에 크게 의존한다.

이 시스템을 실제로 다루는 사람은 **단계에 따라 바뀐다.**

- **셋업 단계**: SEMES 자체 엔지니어가 설치·조정을 진행하며 직접 조치.
- **양산 단계**: 장비는 고객사(삼성전자) 현장에 넘어가고, 이후 알람 대응은 **삼성전자 오퍼레이터**가 1차로 담당.

즉 신입 SEMES 엔지니어뿐 아니라, SEMES 내부 사정을 잘 모르는 고객사 오퍼레이터까지 같은 화면에서 같은 도움을 받아야 한다. 참고할 트러블슈팅 매뉴얼이 부실해 매번 시니어(혹은 SEMES 필드 엔지니어)에게 물어봐야 하고, 과거에 이미 처리된 유사 장애도 다시 처음부터 원인을 찾는 경우가 많다.

이 프로젝트는 **과거 장애 이력(DB) + SOP 문서**를 근거로 삼아, 알람 발생 시 "과거에 이런 알람이 있었고, 이렇게 조치했다"를 자연어로 요약해주는 RAG(Retrieval-Augmented Generation) 기반 어시스턴트를 만드는 것을 목표로 한다.

**VCS의 배차/경로/충돌회피 등 제어 로직 자체는 건드리지 않는다.** 이 프로젝트는 어디까지나 기존 VCS 운영을 보조하는 기능이다 — 안전-critical한 제어 루프에 개입하지 않으므로 1인·저연차 개발자가 완주하기에 현실적인 범위다.

## 2. 운영 환경 및 제약

- **폐쇄망(에어갭) 환경**: 외부 인터넷/외부 AI API 사용 불가. 모든 구성요소는 사내망 안에서 동작.
- AI 모델은 Hugging Face 오픈소스 모델을 다운로드해 로컬 서빙 (이전 스파이크에서 검증한 Ollama + qwen2.5:7b-instruct 조합 재사용 가능).
- 장애 이력은 이미 사내 DB에 쌓여 있음 — 신규로 만들 필요 없이 기존 데이터를 활용하되, 부족한 필드(조치 결과 등)는 보강용 샘플 데이터를 추가해도 됨.
- SOP 문서는 현재 존재하지 않음 — 이 프로젝트 진행 중 직접 작성해 RAG 근거 자료로 사용.
- **실제 UI/App 스택 (기존 VCS)**:
  - **UI**: Java 1.7.0_80 (32bit) + SWT 데스크톱 앱. 구버전이라 REST/API 통신이 사실상 불가 — 지금도 오류 발생 시 App이 DB에 쓴 걸 UI가 **DB 폴링**으로 읽어와 알람을 띄우는 방식.
  - **App**: Java 21 기반 MSA, 총 8개 모듈. API 통신에는 제약이 없음.
  - 따라서 AI 서버 호출처럼 새로운 통신이 필요한 작업은 **App(Java21) 쪽이 수행**하고, UI(Java7)는 지금처럼 DB만 보는 구조를 유지한다 (3번 아키텍처 참고).
- **사용자 역할 관련 결정 사항**:
  - SEMES 담당자와 삼성전자 오퍼레이터는 **같은 알람에 대해 동일한 답변**을 본다 — 역할별로 답변 깊이/내용을 분기하지 않는다. 대신 SOP는 처음부터 고객에게 그대로 노출해도 되는 수준으로만 작성한다 (원가·내부 엔지니어링 근거 같은 SEMES 내부 전용 정보는 포함하지 않음).
  - 이 VCS는 삼성전자 전용 납품 건이라, 장애 이력/SOP를 고객사별로 격리해야 하는 이슈는 없음.

## 3. 시스템 아키텍처

핵심 결정: **운영자(SEMES 셋업 엔지니어든 삼성 오퍼레이터든)는 별도 웹사이트에 들어갈 필요가 없다.** AI 호출은 App(Java21)이 대신 해주고, 결과는 지금 쓰는 DB 폴링 경로 그대로 UI에 얹힌다.

```
        [알람 발생]
             │
             ▼
┌───────────────────────────┐
│  App (Java 21, MSA 8모듈)   │  ── 기존: 오류 발생 시 FAULT 레코드를 DB에 저장
│                             │  ── 신규: 저장 직후 비동기로 AI 서버 호출
└──────────────┬─────────────┘
               │ REST (App은 Java21이라 통신 제약 없음)
               ▼
┌───────────────────────────┐
│  FastAPI AI 서버            │  ── 임베딩 → 벡터 검색(FAULT + SOP) → 로컬 LLM 호출
└──────────────┬─────────────┘        → 구조화 답변을 App에 반환
               │
               ▼
┌───────────────────────────┐
│  App이 진단 결과를 DB에 저장  │  ── FAULT와 1:1 연결 (DIAGNOSIS 레코드)
└──────────────┬─────────────┘
               │ (신규 통신경로 없음 — 기존 폴링 그대로)
               ▼
┌───────────────────────────┐
│  UI (Java 7 / SWT, 32bit)  │  ── 기존 폴링 주기에 알람 + 진단을 함께 조회해 표시
│                             │  ── 피드백(해결됨/안됨)은 UI가 DB에 직접 write (JDBC, 기존 방식)
└───────────────────────────┘
```

**별도 웹 관리자 페이지(React/Next.js)** 는 운영자용이 아니라 **SEMES 내부 전용**으로만 존재 — SOP 작성/수정, 알람 트렌드 확인처럼 실시간성이 없는 작업만 여기서 처리 (9번 기술스택, 5.2.2 참고).

### 3.1 요청 처리 흐름 (End-to-End)

| # | 단계 | 담당 컴포넌트 | 처리 내용 |
|---|---|---|---|
| 1 | 오류 발생 | App (Java21) | 기존과 동일하게 `FAULT` 레코드를 DB에 저장 |
| 2 | AI 진단 요청 | App → AI 서버 | 저장 직후 **비동기**로 AI 서버 호출 (App 자체 오류 처리 흐름을 막지 않기 위함) |
| 3 | 임베딩 + 벡터 검색 | AI 서버 (FastAPI) | 알람코드/메시지 임베딩 → `FAULT` Top-K, `SOP` Top-K 유사도 검색 |
| 4 | 프롬프트 구성 + LLM 호출 | AI 서버 → Ollama | 검색된 컨텍스트 + 알람 정보를 로컬 모델에 전달 (외부 API 없음) |
| 5 | 구조화 답변 생성 | AI 서버 | 원인 / 조치 / 에스컬레이션 기준 / 근거를 구조화 출력. 근거 없으면 "확인 필요" |
| 6 | 진단 결과 저장 | App | AI 서버 응답을 `DIAGNOSIS` 레코드로 DB에 저장, `FAULT`와 연결 |
| 7 | 화면 표시 | UI (Java7/SWT) | **기존 폴링 주기**에 알람 + 진단을 함께 조회해 팝업/패널에 표시 (신규 통신 없음) |
| 8 | 피드백 | UI → DB | 운영자가 "해결됨/안됨"을 남기면 UI가 DB에 직접 write (기존 JDBC 쓰기 경로 재사용) |

```
1) INSERT INTO FAULT (id, amr_id, alarm_code, message, occurred_at)      -- 기존 동작

2) (App, 비동기) POST http://ai-server/diagnose  { "faultId": "FAULT-2025-114" }

3) embed(alarm) → top_k(FAULT, k=5)  +  top_k(SOP, k=3)

4) model: qwen2.5:7b-instruct  ·  context: FAULT×5 + SOP×3

5) { "cause": [...], "action": [...], "escalation": "...", "evidence": ["FAULT-2025-089", "SOP-007"] }

6) INSERT INTO DIAGNOSIS (fault_id, cause, action, escalation, evidence)

7) (UI, 기존 폴링) SELECT * FROM FAULT f JOIN DIAGNOSIS d ON f.id = d.fault_id WHERE ...

8) UPDATE DIAGNOSIS SET feedback = 'resolved' WHERE fault_id = ...        -- UI가 직접 write
```

### 3.2 화면 와이어프레임 (SWT 알람 팝업 / 패널)

핵심은 답변 자체보다 **근거를 눈에 보이게 붙여두는 것**과, **어디까지 직접 조치하고 어디서부터 SEMES를 불러야 하는지 명시하는 것** — LLM 답변을 그대로 믿지 않고 원본 사례와 바로 대조할 수 있게 하고, 오퍼레이터가 권한 밖의 조치까지 시도하지 않게 한다.

```
┌────────────────────────────────────────────────────────────┐
│ [알람] AL-4821 · 3번 AMR · Stocker A 구간                     │
├────────────────────────────────────────────────────────────┤
│ 가능한 원인                                                   │
│  · Stocker A 진입 구간 위치 센서 오차 누적 (유사 사례 2건)     │
│  · 해당 구간 정체로 인한 안전정지 트리거                       │
│                                                                │
│ 조치 (직접 가능)                                              │
│  1. AMR 위치 캘리브레이션 재실행                                │
│  2. Stocker A 진입 센서 청소 상태 확인                         │
│                                                                │
│ ⚠ 위 조치로 해결 안 되면 → SEMES 필드 엔지니어 호출 (SOP-007)  │
│                                                                │
│ 참고 사례   [FAULT-2025-089]  [FAULT-2025-220]  [SOP-007]     │
├────────────────────────────────────────────────────────────┤
│ 조치 후 해결됐나요?                      [해결됨]  [해결 안 됨] │
│ 근거 3건 · 자동 진단 (알람 발생 4초 후 표시)                    │
├────────────────────────────────────────────────────────────┤
│ 추가로 설명하고 싶은 상황이 있나요? [________________] [질문]  │
└────────────────────────────────────────────────────────────┘
```

- 상단 답변 영역은 알람 저장 직후 App이 이미 진단까지 받아 DB에 넣어둔 상태라, 오퍼레이터가 팝업을 열었을 때 **이미 채워져 있다** (5.1.1/A).
- "조치(직접 가능)"와 "⚠ 에스컬레이션 기준"을 분리 표기 — 오퍼레이터/SEMES 구분 없이 누가 보든 동일하지만, 권한 밖 조치와 SEMES 호출 시점은 명확히 구분 (5.1.1 갱신 내용).
- 맨 아래 자유 텍스트 입력은 알람코드만으로 설명이 부족할 때 쓰는 보조 경로(B 기능) — 주 경로는 타이핑 없이 자동으로 채워짐.

## 4. 데이터 모델

| ID | 설명 |
|---|---|
| `FAULT-{연도}-{번호}` | 개별 장애/알람 이력. 기존 DB 데이터 재사용 |
| `SOP-{번호}` | SOP 문서를 조항 단위로 쪼갠 청크. 고객에게 노출 가능한 수준으로만 작성 |
| `DIAGNOSIS-{번호}` | `FAULT` 1건당 자동 생성되는 AI 진단 결과 + 오퍼레이터 피드백 (신규 테이블, 1:1) |
| `QUERY-{번호}` | 자유 텍스트로 추가 질문한 경우의 질의응답 이력 (보조 경로, B 기능 전용) |

**FAULT** 주요 필드: 발생시각, AMR ID, 알람코드, 알람메시지, 발생 구간(예: Stocker A ↔ Prober B), 조치 내용, 조치자, 해결까지 걸린 시간

**SOP** 주요 필드: 제목, 카테고리(알람코드 그룹), 본문, 관련 알람코드 리스트

**DIAGNOSIS** 주요 필드: 연결된 `FAULT-ID`, 원인 후보(cause), 직접 조치 가능 항목(action), 에스컬레이션 기준(escalation), 근거 `FAULT`/`SOP` ID 목록(evidence), 생성 시각, 오퍼레이터 피드백(해결됨/안됨) — 5.2.1(피드백 루프)의 기반 데이터

**QUERY** 주요 필드: 질의 원문, 검색된 근거 ID 목록, LLM 응답, 질의 시각

## 5. 기능 목록

### 5.1 핵심 모듈 (Must-have)

| # | 기능 | 설명 |
|---|---|---|
| A | 알람 자동 진단 (RAG) | 알람 발생 즉시 App이 AI 서버를 호출, 유사 과거 장애(FAULT) + SOP를 검색해 원인·조치·에스컬레이션 기준을 자동 생성. 오퍼레이터의 타이핑 없이 팝업에 바로 표시. 상세 5.1.1 |
| B | 자연어 추가 질의 (보조) | 알람코드만으로 부족할 때 자유 텍스트로 상황을 추가 설명하면 동일 파이프라인 사용. 상세 5.1.2 |
| C | 근거 출처 표시 | 답변과 함께 참고한 FAULT-ID/SOP-ID를 표시해 신뢰성·추적성 확보 (환각 방지 핵심 장치). 상세 5.1.3 |

#### 5.1.1 상세 — A. 알람 자동 진단 (RAG)

- 트리거: 사람이 입력하는 게 아니라 **App이 FAULT 레코드를 저장하는 시점에 자동으로 발생**.
- 처리: 알람코드/메시지를 임베딩 → 벡터 유사도로 FAULT 이력 Top-K, SOP 청크 Top-K 검색 → 검색된 컨텍스트를 LLM에 넣어 "가능한 원인 / 오퍼레이터가 직접 할 수 있는 조치 / SEMES 호출이 필요한 에스컬레이션 기준 / 참고할 만한 과거 사례"로 구조화 요약.
- **조치와 에스컬레이션을 분리하는 이유**: 양산 단계에서는 삼성전자 오퍼레이터가 답변을 그대로 실행하게 되므로, SEMES 필드 엔지니어가 아니면 하면 안 되는 조치까지 안내하면 위험하다. SOP 작성 시점부터 "오퍼레이터가 해도 되는 범위"와 "SEMES를 불러야 하는 조건"을 구분해 태깅해 둔다.
- 과거 사례가 전혀 없는 신규 알람이면 "유사 사례 없음"을 명시하고 SOP만으로 답변하거나, 답변 불가를 솔직히 표시 (없는 사례를 지어내지 않는 것이 핵심 요구사항).

#### 5.1.2 상세 — B. 자연어 추가 질의 (보조)

- 자동 진단(A)만으로 상황이 애매할 때, 오퍼레이터가 자유 텍스트로 추가 설명("멈췄다가 다시 움직였다가 반복해요" 등)을 입력할 수 있는 보조 입력창을 SWT 패널 하단에 둔다.
- 자유 텍스트 → (필요시) 관련 알람코드 후보를 LLM이 먼저 추정 → 그 알람코드로 A와 동일한 검색/요약 수행.
- 이 경로는 `QUERY` 테이블에 별도 기록 (자동 진단과 구분).

#### 5.1.3 상세 — C. 근거 출처 표시

- 답변 화면에 "이 답변은 FAULT-2025-089, FAULT-2025-220, SOP-007을 근거로 생성됨"처럼 원본 링크를 함께 노출.
- 담당자가 원문(원래 장애 이력 상세, SOP 원문)을 즉시 대조 확인할 수 있게 함 — LLM 답변을 그대로 믿지 않고 검증 가능하게 만드는 것이 목적.

### 5.2 확장 기능

| # | 기능 | 설명 |
|---|---|---|
| 1 | 조치 결과 피드백 루프 | 답변대로 조치한 결과(해결/미해결)를 남기면 향후 검색 랭킹에 반영. 상세 5.2.1 |
| 2 | SOP 문서 관리 (SEMES 내부 전용 웹) | SEMES 담당자가 SOP를 추가/수정하고 재임베딩. 오퍼레이터는 접근하지 않음. 상세 5.2.2 |
| 3 | 알람 트렌드 대시보드 (SEMES 내부 전용 웹) | 최근 급증한 알람, 반복 발생 Top N 집계. 상세 5.2.3 |
| 4 | 장애 처리 후 SOP 초안 자동 생성 | 신규 장애의 원인/조치 기록을 LLM이 SOP 초안 형태로 요약 제안. 상세 5.2.4 |
| 5 | 유사 알람 클러스터링 | 과거 장애 이력을 임베딩 기반으로 군집화해 미분류 패턴 발견. 상세 5.2.5 |

#### 5.2.1 상세 — 1번. 조치 결과 피드백 루프

- 답변을 본 운영자가 SWT 패널에서 "이 조치로 해결됨" / "해결 안 됨"을 누르면, UI가 DB에 직접 write(JDBC, 기존 방식) → `DIAGNOSIS.feedback` 갱신.
- 해결됨으로 표시된 사례는 이후 유사 검색에서 가중치를 높여 우선 노출 (단순 최신순보다 "실제로 통했던 사례" 우선).

#### 5.2.2 상세 — 2번. SOP 문서 관리 (SEMES 내부 전용 웹)

- SOP는 이 프로젝트 기간 중 처음부터 작성해야 하므로, SEMES 담당자가 웹에서 SOP를 추가/수정할 수 있는 화면이 필요. 이 화면은 오퍼레이터가 아니라 SEMES 담당자만 쓰는 내부 관리자 도구.
- 저장 시 자동으로 청크 분할 + 재임베딩해 검색 대상에 즉시 반영.
- 작성 가이드: "고객에게 그대로 노출된다"는 전제로 작성 — 내부 원가/엔지니어링 근거 상세는 포함하지 않고, 조치 항목마다 오퍼레이터 수행 가능 여부(직접 가능/SEMES 호출)를 태깅.

#### 5.2.3 상세 — 3번. 알람 트렌드 대시보드 (SEMES 내부 전용 웹)

- 최근 N일 알람 발생 빈도를 알람코드/AMR/구간별로 집계.
- 급증한 알람이나 반복 발생 알람을 하이라이트해 "SOP 우선 작성 대상"을 자연스럽게 추천하는 효과도 있음 (4번과 연결).

#### 5.2.4 상세 — 4번. 장애 처리 후 SOP 초안 자동 생성

- 신규 장애를 처리한 SEMES 담당자가 원인/조치를 자유 텍스트로 기록하면, LLM이 이를 SOP 포맷(제목/카테고리/본문/조치 vs 에스컬레이션 구분/관련 알람코드)의 초안으로 변환 제안.
- SOP가 아예 없는 상태에서 시작하는 프로젝트 특성상, SOP 자산을 수작업 없이 축적하는 핵심 장치.
- 초안은 자동 확정하지 않고 **SEMES 담당자** 검토 후 2번 화면에서 확정 (AI 결과는 초안이라는 원칙 — 고객에게 노출되는 내용이므로 확정 권한은 SEMES에게만 있음).

#### 5.2.5 상세 — 5번. 유사 알람 클러스터링

- 과거 FAULT 이력을 임베딩 기반으로 군집화해, 알람코드는 다르지만 실제로는 같은 근본 원인을 공유하는 그룹을 찾아냄.
- SOP 미작성 상태에서 "이런 알람 그룹부터 SOP를 만들면 커버리지가 크게 오른다"는 우선순위 판단 근거로 활용.

## 6. 비기능 요구사항

- 폐쇄망 환경에서 전 구성요소 동작 (모델/패키지는 사전 다운로드 후 반입).
- UI(Java7/SWT)는 AI 서버와 직접 통신하지 않으므로, 구버전 JDK의 TLS/HTTP 클라이언트 제약은 이 설계에서는 문제가 되지 않는다 — App(Java21)만 AI 서버와 통신.
- 실제 알람 대응 상황에서 쓰이므로 App→AI 서버 진단 지연이 크면 안 됨 — 목표 응답 시간(예: 알람 발생 후 수 초 이내 진단 표시)을 프로토타입 단계에서 실측 후 확정. 비동기 처리이므로 진단이 늦어도 알람 자체 표시는 지연되지 않음.
- AI 서버는 GPU 자원이 제한적인 공용 워크스테이션에서도 동작 가능하도록 경량/양자화 모델을 기본 옵션으로 고려 (이전 스파이크의 qwen2.5:7b-instruct 실측치 재사용).
- LLM이 근거 없는 답변을 지어내지 않도록(환각) 프롬프트/검증 장치 필요 — 근거(FAULT/SOP ID)가 하나도 검색되지 않으면 "확인 필요" 답변으로 유도.
- SOP/답변 콘텐츠는 역할 구분 없이 고객사(삼성전자)에 그대로 노출된다는 전제로 작성 — 내부 전용 정보를 담지 않는다.

## 8. 향후 진행 순서 (제안)

1. 장애 이력 DB 샘플 추출 + SOP 5~10건 직접 작성 (우선순위 높은 알람 위주, 조치/에스컬레이션 구분 태깅 포함)
2. AI 모듈(A) 스파이크 — 소규모 샘플로 RAG 파이프라인(임베딩→검색→LLM 요약)이 실제로 쓸만한 답변을 주는지 검증 (이전 프로젝트 스파이크 방식 재사용)
3. 데이터 모델/스키마 확정 (`DIAGNOSIS` 테이블 포함), App 쪽 AI 서버 호출 연동 지점 설계
4. UI(SWT) 와이어프레임 — 기존 알람 팝업에 진단 영역 추가
5. SEMES 내부 전용 관리자 웹(SOP 관리·트렌드) 최소 구현
6. 확장 기능(1, 4, 5) 순차 적용, 피드백 루프(1번)부터 우선
