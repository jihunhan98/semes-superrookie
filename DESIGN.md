# 고객 요구사항 명세서 자동화 시스템 — 설계서

## 1. 개요

SEMES는 고객사(삼성전자)로부터 요구사항 명세서(docx/텍스트)를 받아 개발을 진행한다. 이때 **명세서에 모호·다의적 표현이 많아 사람마다 다르게 해석하는 문제**와, **요구사항으로부터 개발할 기능을 도출·추적하는 수작업 문제**가 반복된다.

이 시스템은 요구사항 원문을 입력받아 다음 두 가지를 자동화한다.

```
요구사항 원문(docx)
      │
      ▼
[핵심 기능 1] 모호성 해결      ← 모호한 조항을 찾아내고, 합의를 거쳐 해석을 확정
      │
      ▼
[핵심 기능 2] 기능 요구사항 도출 ← 확정된 요구사항을 실제 개발 "기능" 단위로 분해
      │
      ▼
(추후) 기능 추적 · 개발 완료 판별 · 대시보드 …  → 4장 "추후 개발 추가기능"
```

**이 문서의 범위**: 지금은 위 두 핵심 기능만 상세히 설계한다. 그 외 기능은 4장에 목록으로만 두고, 데이터 모델/ID 체계는 두 핵심 기능의 상세가 굳은 뒤 확정한다(현 단계 보류).

### 1.1 운영 환경 및 제약

- **폐쇄망 환경**: 외부 인터넷/외부 AI API 사용 불가. 전 구성요소가 사내망 안에서 동작.
- AI 모델은 사내 워크스테이션(GPU 사양 미확인)에서 Hugging Face 오픈소스 모델을 로컬 서빙.
- docx 파싱은 표·이미지가 포함된 실제 명세서 포맷을 견고하게 처리해야 함.

### 1.2 아키텍처 (요약)

React 프론트엔드 ─REST─ Spring Boot 백엔드 ─REST─ Python(FastAPI) AI 추론 서버 ─ Hugging Face 로컬 LLM.
백엔드는 Oracle DB에 상태를 저장하고, AI 서버는 무상태(입력 → 구조화 JSON 출력)로 유지한다.

## 2. 핵심 기능 1 — 모호성 해결

모호한 조항을 **① 찾아내고(검출) → ② 합의를 거쳐 해석을 확정(해결)** 하는 것까지가 이 기능의 범위다.
원래 문제인 "부서마다 다른 해석"을 직접 푸는 핵심 기능.

### 2.1 모호성 유형 분류 (taxonomy)

"모호함"을 사람이 아니라 AI가 일관되게 판정하려면 기준이 되는 유형 분류가 필요하다.

| 유형 | 설명 | 예시 문장 |
|---|---|---|
| 정량 기준 부재 | 수치·기준값 없이 정성적으로만 표현 | "장비는 충분한 내구성을 가져야 한다" |
| 모호한 정도부사 | 사람마다 기준이 다르게 해석되는 부사 | "적절히 조정한다", "가능한 한 빠르게 처리한다" |
| 주어/주체 불명확 | 누가 수행하는지 명시되지 않음 | "관련 부서와 협의하여 처리한다" |
| 조건 발생 시점 불명확 | 트리거 조건이 정의되지 않음 | "필요한 경우 개선한다" |
| 예외/경계 조건 누락 | 정상 상황만 정의, 예외 케이스 없음 | "정상 동작해야 한다" |
| 접속사 범위 모호 | and/or 처리 범위가 불명확 | "A 및/또는 B를 지원한다" |
| 시간·일정 모호 | 구체적 기한 없이 표현 | "빠른 시일 내 대응한다", "적시에 통보한다" |

### 2.2 검출

- 입력: docx/텍스트 요구사항을 조항 단위로 파싱.
- 각 조항을 AI에 전달해 모호 여부를 판정하고, 모호하면 위 유형(enum)과 근거 문장을 함께 출력한다.
- "모호함"이라고만 표시하지 않고 **어떤 유형인지까지** 출력해 담당자가 빠르게 맥락을 파악하게 한다.
- 근거가 없으면(판정 불가) 지어내지 않고 그대로 "판정 불가"로 남긴다.

> _(이 절의 상세 규칙 — 파싱 단위, 출력 스키마, 프롬프트 전략 등 — 은 여기에 이어서 채운다.)_

### 2.3 해결 (합의·확정 흐름)

- 모호로 판정된 조항마다 코멘트 스레드가 생성된다.
- 설계/품질/영업 등 관련 담당자가 각자 해석을 코멘트로 병렬 등록한다.
- 지정된 최종 확정자(PM/리더)가 여러 해석 중 하나를 "확정"으로 지정한다.
- 확정되면 해당 조항은 잠금(lock) 처리된다.

> _(확정 권한, 잠금 규칙, 미해결 조항 추적 방식 등 상세는 여기에 이어서 채운다.)_

### 2.4 입출력 예시

실제 요구사항([PROJECT.md](PROJECT.md) 7.1 "태스크 전개와 전환")을 이 기능에 통과시키면 다음과 같이 처리된다.

**입력 (원문 조항):**
> 태스크 각 스텝별 진행에 필요한 **필수 요건이 충족된 상태인 경우에 한하여**, AMR 사용자가 메뉴를 통해 **몇 단계 이전** 스텝으로 돌아가거나 **몇 단계를 건너뛴** 스텝에서 태스크를 다시 전개할 수 있어야 한다.

**검출 결과:**

| 근거 표현 | 유형 | 사유 |
|---|---|---|
| "필수 요건이 충족된 상태" | 정량 기준 부재 / 조건 불명확 | "필수 요건"이 무엇인지 정의되지 않음 — 스텝마다 어떤 조건을 만족해야 하는지 불명확 (원문 비고에서도 "정의해야 한다"고 명시) |
| "몇 단계 이전 / 몇 단계를 건너뛴" | 정량 기준 부재 | 되돌아가거나 건너뛸 수 있는 단계 수의 범위가 없음 — 1단계만인지, 임의 단계인지 불명확 |

```json
{
  "req_id": "REQ-2026-041",
  "ambiguous": true,
  "findings": [
    { "span": "필수 요건이 충족된 상태", "type": "정량 기준 부재", "reason": "..." },
    { "span": "몇 단계 이전 / 몇 단계를 건너뛴", "type": "정량 기준 부재", "reason": "..." }
  ]
}
```

**해결 후 (합의로 명확화된 예시):**
> 각 스텝의 **필수 요건**은 [스텝별 요건표]로 정의한다. 사용자는 **직전 1개 스텝** 또는 **현재 스텝 기준 최대 2단계 뒤** 스텝에서만 태스크를 다시 전개할 수 있다.

> _(모호성 유형별 "명확화 가이드" — 어떤 유형이면 무엇을 물어 채워야 하는지 — 를 여기에 이어서 정리. 예시는 사용자가 추가로 제공하는 케이스로 계속 보강한다.)_

## 3. 핵심 기능 2 — 기능 요구사항 도출

모호성이 해소된(또는 애초에 모호하지 않은) 조항을, AI가 실제 개발 단위인 **"기능"** 으로 분해한다.

### 3.1 분해

- 요구사항 문장을 개발 가능한 기능 단위로 분해한다.
- 요구사항 1개가 여러 기능으로 나뉘거나, 여러 요구사항이 하나의 기능으로 합쳐지는 다대다 관계를 허용한다.

> _(분해 기준, 기능 1건의 표현 형식, 요구사항↔기능 연결 방식 등 상세는 여기에 이어서 채운다. 연결 테이블/ID 체계는 데이터 모델 확정 시점으로 미룸 — 4장.)_

### 3.2 검토·수정 (초안 원칙)

- AI가 자동 추출한 기능 목록을 그대로 확정하지 않고, 담당자가 검토·수정할 수 있는 화면을 제공한다.
- **AI 결과는 항상 초안**이라는 원칙 (모호성 검출·해결과 동일).

### 3.3 입출력 예시

> _(핵심 기능 1의 2.4처럼, 실제 요구사항 한 건이 어떤 기능들로 분해되는지 예시를 여기에 채운다.)_

## 4. 추후 개발 추가기능 (상세 설계는 이후 단계)

두 핵심 기능이 자리 잡은 뒤 순차적으로 붙인다. 지금은 목록만 유지한다.

| 기능 | 한 줄 설명 |
|---|---|
| 요구사항 개정판 비교(diff) | 새 버전 명세서를 이전 버전과 조항 단위로 비교해 변경/추가/삭제 자동 탐지 |
| 모호성 해결 이력(감사 추적) | 확정된 해석 + 시점 + 확정자 + 원래 모호 사유를 영구 이력으로 보관 |
| 요구사항 ↔ 기능 연결 | 요구사항과 기능을 양방향으로 연결(link)해 서로 조회 |
| Bitbucket 연동 개발완료 판별 | 커밋/PR의 기능 태그를 감지해 개발 진행 상태 자동 갱신 |
| 진행 현황 대시보드 (SCCB) | 기능 상태를 SCCB 프로세스 단계별로 집계·시각화 |
| 내부 발의 기능 관리 | 고객 요구사항과 무관하게 사내에서 발의하는 기능 등록·관리 |
| **데이터 모델 / ID 체계 확정** | 요구사항·기능·연결·이력 테이블과 ID 체계 설계 — **두 핵심 기능 상세가 굳은 뒤 착수** |
